核心概念

DataStream

Flink 的核心数据抽象，表示元素持续产生的无界数据流。

Transformation

对 DataStream 进行操作：map、filter、keyBy、window、reduce 等。

Stateful

有状态流处理：聚合、连接等需要维护中间结果的运算。

Sink

数据输出到 Kafka、ES、Redis、JDBC、HDFS 等外部系统。

Slot

TaskManager 内的资源单元，一个 Slot 可执行多个 subtask。

Operator Chain

将多个算子链接为一个 task 执行，减少线程切换与序列化开销。

时间语义

Flink 支持三种时间类型，推荐在乱序场景下使用 Event Time

Event Time

事件时间

数据产生的时间，业务时间。处理乱序/延迟数据时推荐使用。

Processing Time

处理时间

机器处理数据时的本地时间，性能最高但结果不确定。

Ingestion Time

摄入时间

数据进入 Flink 的时间，介于 Event Time 与 Processing Time 之间。

Window 窗口

将无界流切分为有界的数据块进行处理，必须在 KeyBy 之后使用

滚动窗口

Tumbling Window

固定大小、不重叠、无间隙。如每 5 秒统计一次。

滑动窗口

Sliding Window

固定大小、可重叠、有滑动步长。如每 5 秒统计最近 10 秒。

会话窗口

Session Window

根据活动间隙动态划分，适合用户行为分析等场景。

全局窗口

Global Window

统一处理所有数据，需配合自定义触发器使用。

Checkpoint 机制

基于 barrier 的分布式快照，实现 Exactly-once 状态化容错

1. JobManager 触发

周期性向 Source 发送 barrier

→

2. Source 快照

记录消费 offset 后向下游传递 barrier

→

3. 算子快照状态

每个算子收到 barrier 后保存当前状态

→

4. Checkpoint 完成

所有算子汇报完成，通知 JobManager

Structured Streaming

Spark 生态中的实时流处理组件：把流式数据看作不断增加的表

核心抽象

无界输入表（Unbounded Input Table）：流式数据 = 不断增加的数据库表。

Result Table

对输入表执行标准 SQL 查询，每次查询产生的结果持续更新。

三种输出模式：Complete 输出完整结果集，Append 仅追加新增，Update 仅输出变化

Complete Mode

完整模式

输出完整结果集，适用于聚合查询。

Append Mode

追加模式

仅输出新增数据，适用于非聚合查询。

Update Mode

更新模式

仅输出更新/变化的数据，适用于有更新的查询。
